# Правила анализа гипотез и идей

Этот файл описывает, как агент должен проверять идеи, лежащие в папке `hypotheses/`.

---

## 1. Формат входных файлов

Файл-идея — произвольный `.md` с описанием гипотезы, наброском доказательства, ссылками на литературу. Имя файла содержит краткое название темы и дату создания. Никаких жёстких требований к структуре: автор пишет так, как ему удобно.

## 2. Формат файла-ответа

Агент создаёт файл с именем:

```
<Модель>_Answer_<имя_исходного_файла>.md
```

Например: `Opus4.6_Answer_Графы_pancake_и_остовы_асимптотика_2026_03_04_06_06_04.md`.

### Обязательная структура ответа

1. **Метаданные** — дата анализа, модель, ссылка на исходный файл.
2. **Резюме идеи** (2–5 предложений) — суть того, что предлагается.
3. **Вердикт** — одна строка: `ПОДТВЕРЖДЕНА` / `ЧАСТИЧНО ВЕРНА` / `ТРЕБУЕТ ДОРАБОТКИ` / `ОШИБОЧНА` / `НЕ ПРОВЕРЯЕМА` (с коротким пояснением).
4. **Детальный разбор** — по каждому утверждению / гипотезе из исходного файла:
   - Формулировка утверждения (кратко).
   - Оценка корректности (верно / неверно / неточно / тривиально / нетривиально).
   - Обоснование: ссылки на теоремы, статьи, код проекта или контрпримеры.
5. **Связь с проектом** — как идея может (или не может) быть использована в коде, экспериментах, статье. Ссылки на конкретные модули (`src/...`), гипотезы (`docs/03_HYPOTHESES.md`), результаты (`docs/04_RESULTS.md`).
6. **Рекомендации** — что стоит доработать, проверить вычислительно, добавить в статью или в код.

## 3. Порядок работы агента

### Подготовка

1. Прочитать документацию проекта: `docs/00_AGENT_NAVIGATION.md` → `01_PROJECT_OVERVIEW.md` → `03_HYPOTHESES.md` → `04_RESULTS.md` → `06_REFACTORING_PLAN.md`. Убедиться, что понимаешь контекст задачи, текущий код и существующие гипотезы.
2. Прочитать файл-идею целиком.
3. При необходимости — поискать в интернете релевантные статьи, теоремы, контрпримеры. Не ограничиваться только тем, что есть в проекте.

### Анализ

4. Для каждого утверждения в файле-идее:
   - Проверить математическую корректность (определения, формулировки, доказательства).
   - Проверить, не является ли утверждение тривиальным следствием известного результата (и если да — указать какого).
   - Проверить, нет ли ошибок в рассуждениях (пропущенные случаи, неверные оценки, некорректное применение теорем).
   - Оценить новизну: известен ли результат в литературе.
5. Оценить практическую применимость к проекту:
   - Может ли идея привести к улучшению солверов, эвристик, beam search?
   - Может ли идея войти в статью?
   - Нужна ли вычислительная проверка?

### Запись

6. Написать файл-ответ по структуре из раздела 2.
7. Быть честным: если идея ошибочна — объяснить, где и почему. Если идея верна, но тривиальна — указать это. Если не уверен — так и написать с указанием, что нужно для окончательного вывода.

## 4. Принципы

- **Доказательность.** Каждая оценка подкреплена ссылкой (статья, теорема, код, вычисление) или помечена как мнение.
- **Честность.** Не подтверждать идею из вежливости. Не отвергать без обоснования. Цель — помочь автору довести идею до публикабельного состояния или отбросить нерабочую.
- **Связь с проектом.** Даже чисто теоретическая идея оценивается в контексте pancake-проекта: помогает ли она улучшить скор, объясняет ли поведение алгоритмов, даёт ли новые гипотезы для экспериментов.
- **Конкретность.** Рекомендации содержат конкретные шаги (какой эксперимент запустить, какую теорему проверить, какой код написать), а не общие пожелания.
- **Минимальность.** Не раздувать ответ. Если утверждение очевидно верно — достаточно одного предложения. Развёрнутый разбор — только для спорных или ошибочных мест.
